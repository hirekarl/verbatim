# Bootstrapping Verbatim

This is a runbook for scaffolding professional Python tooling, CI/CD, and GitHub
governance for the Verbatim repo. It is a **plan to execute**, not a description of
work already done — as of writing, this repo contains only PRD docs, a
`brand_guidelines.json` fixture, and a standalone `brand_guidelines.py` prototype. No
package structure, tests, CI, or GitHub remote exist yet.

Read the whole document once before running anything — the phases are ordered on
purpose (e.g. branch protection can't be configured before the GitHub repo exists, and
the first commit message matters for the first automatic release).

## What's already fine as-is

- **`.gitignore`** — the existing file is a solid macOS/Windows/Python composite
  (already ignores `.venv`, `.mypy_cache`, `.ruff_cache`, `.coverage`, etc.). Do not
  regenerate it.
- **`brand_guidelines.py` / `brand_guidelines.json`** — stay at the repo root for this
  bootstrap. See "Follow-up" at the end of this document for their intended future
  home; that migration is explicitly out of scope here.

## Decisions this plan assumes (already made, don't re-litigate)

| Area | Decision |
| --- | --- |
| Python version | 3.12, fully managed by **uv**; `uv.lock` is committed |
| Repo visibility/license | Public GitHub repo, **MIT** license |
| Branch protection | 1 required approving review + required passing CI on `main`; admin (Karl) can bypass with `--admin` |
| Versioning | **Fully automatic** semver bump + changelog + tag + GitHub Release via commitizen on every merge to `main` |
| Dependencies | Dev-only for now: ruff, mypy, pytest, pytest-cov, pre-commit, commitizen. No product deps (Google API client, LLM SDK, etc.) yet |
| Onboarding | README-only, detailed macOS + Windows steps — no devcontainer/Docker |
| Task runner | Plain `uv run ...` commands documented in the README — no Makefile/justfile |
| Coverage gate | CI fails if coverage drops below **90%** |
| Process norm | **TDD** — write a failing test before implementation code (documented, not tool-enforced) |

## File-by-file specification

### `pyproject.toml`

Single source of truth for project metadata and all tool configuration.

- `[project]`: `name = "verbatim"`, `version = "0.0.0"` (deliberately starts at zero —
  see Phase 3), `requires-python = ">=3.12,<3.13"`, `license = {text = "MIT"}`,
  `authors = [{name = "Karl Johnson"}, {name = "Christina Ruiz"}]`, `readme = "README.md"`.
- `[build-system]`: `requires = ["hatchling"]`, `build-backend = "hatchling.build"`.
- `[tool.hatch.build.targets.wheel]`: `packages = ["src/verbatim"]`.
- `[dependency-groups]`: `dev = ["ruff", "mypy", "pytest", "pytest-cov", "pre-commit", "commitizen"]`
  (written automatically by `uv add --dev ...`).
- `[tool.ruff]`: `target-version = "py312"`, `line-length = 88`.
  - `[tool.ruff.lint]`: `select = ["E", "F", "W", "I", "N", "UP", "B", "C4", "SIM", "D", "RUF"]`.
  - `[tool.ruff.lint.pydocstyle]`: `convention = "google"`.
  - `[tool.ruff.lint.per-file-ignores]`: `"tests/**" = ["D"]` (relax docstring rules in tests).
- `[tool.mypy]`: `python_version = "3.12"`, `strict = true`, `disallow_untyped_defs = true`,
  `disallow_incomplete_defs = true`, `warn_unused_ignores = true`,
  `warn_redundant_casts = true`, `no_implicit_optional = true`, `show_error_codes = true`,
  `files = ["src", "tests"]`. Strict mode applies to tests too — test functions need
  `-> None` return annotations.
- `[tool.pytest.ini_options]`: `testpaths = ["tests"]`,
  `addopts = "--cov=src/verbatim --cov-report=term-missing --cov-report=xml --cov-fail-under=90"`.
- `[tool.coverage.run]`: `source = ["src/verbatim"]`, `branch = true`.
- `[tool.coverage.report]`: `exclude_lines = ["pragma: no cover", "if TYPE_CHECKING:", "raise NotImplementedError"]`.
- `[tool.commitizen]`: `name = "cz_conventional_commits"`, `version = "0.0.0"`,
  `version_provider = "pep621"`, `version_scheme = "semver"`,
  `update_changelog_on_bump = true`, `tag_format = "v$version"`, `major_version_zero = true`
  (keeps `feat:` bumping the minor version while pre-1.0, avoiding a premature `1.0.0`).

### `src/verbatim/__init__.py`

Single-sources `__version__` via
`from importlib.metadata import version; __version__ = version("verbatim")` — avoids a
second hardcoded version string that commitizen would otherwise need a `version_files`
entry to keep in sync with `pyproject.toml`.

### `src/verbatim/py.typed`

Empty file. PEP 561 marker declaring the package ships inline types.

### `tests/test_version.py`

Asserts `verbatim.__version__` is a non-empty string. This seeds the 90% coverage gate
meaningfully from the very first CI run — an empty `src/` tree would otherwise make
`pytest-cov` error ("no data to report") or trivially pass on zero statements.

### `.pre-commit-config.yaml`

- Top-level `default_install_hook_types: [pre-commit, commit-msg]` so a single
  `uv run pre-commit install` wires both hook stages — important so a non-professional
  collaborator doesn't need to remember extra flags.
- `pre-commit/pre-commit-hooks`: `trailing-whitespace`, `end-of-file-fixer`,
  `check-yaml`, `check-toml`, `check-merge-conflict`, `mixed-line-ending`
  (`args: [--fix=lf]`), `check-added-large-files` (**`args: [--maxkb=2000]`** — must be
  raised above the 500 KB default because `Verbatim PRD.docx` is 886 KB and would
  otherwise fail this hook on every touch).
- `astral-sh/ruff-pre-commit`: `ruff` (`--fix`) and `ruff-format`.
- `pre-commit/mirrors-mypy`: `mypy`.
- `commitizen-tools/commitizen`: hook id `commitizen`, `stages: [commit-msg]`.
- `gitleaks/gitleaks`: default ruleset. **Chosen over detect-secrets** — a single
  static binary with no `.secrets.baseline` file for a non-developer collaborator to
  generate or audit, and its default rules already cover Google API key/OAuth secret
  patterns relevant to this project's eventual Google Docs API integration.

### `.github/workflows/ci.yml`

- `name: CI`; triggers on `pull_request` and `push: branches: [main]`.
- Job `Quality Gates`: `astral-sh/setup-uv@v5` (`enable-cache: true`),
  `uv python install 3.12`, `uv sync`, then `uv run ruff check .`,
  `uv run ruff format --check .`, `uv run mypy src`, `uv run pytest`.
- Job `pr-title-lint` (runs only `on: pull_request`): `amannn/action-semantic-pull-request@v5`
  to validate the PR title against Conventional Commits — required because of the
  squash-merge decision in Phase 4.

### `.github/workflows/release.yml`

- Triggers on `workflow_run` for the CI workflow, filtered to
  `github.event.workflow_run.conclusion == 'success'` and `head_branch == 'main'` —
  guarantees a release only happens after CI genuinely passed on `main`.
- Checks out and authenticates using a dedicated **`RELEASE_PAT`** secret (not the
  default `GITHUB_TOKEN` — see Phase 4 for why this is required).
- Runs `commitizen-tools/commitizen-action` (bump version + changelog + tag + push).
- Publishes the GitHub Release via `softprops/action-gh-release@v2`, referencing the
  new tag and the freshly generated changelog section.

## Branch protection: exact commands + why

**Use classic branch protection, not Repository Rulesets.** "Bypass via `--admin`"
maps directly to the classic `enforce_admins` boolean; Rulesets would need a
`bypass_actors` role-ID array for the same effect — more moving parts for no benefit at
this repo's scale (2 collaborators, one branch).

```bash
gh api repos/<owner>/verbatim/branches/main/protection \
  --method PUT \
  --input - <<'EOF'
{
  "required_status_checks": {
    "strict": true,
    "contexts": ["Quality Gates", "pr-title-lint"]
  },
  "enforce_admins": false,
  "required_pull_request_reviews": {
    "required_approving_review_count": 1,
    "require_code_owner_reviews": true,
    "dismiss_stale_reviews": true
  },
  "restrictions": null,
  "required_linear_history": true,
  "allow_force_pushes": false,
  "allow_deletions": false
}
EOF
```

**Caveat:** the exact `contexts` strings are the check-run names GitHub Actions
reports, which can't be known with certainty until after the first workflow run. Run
this command *after* the first CI run on `main`, verifying the real names via
`gh pr checks` or `gh api repos/<owner>/verbatim/commits/main/check-runs` first.

Also configure squash-only merges with the PR title as the commit message (required so
commitizen reads real, semver-meaningful history):

```bash
gh api repos/<owner>/verbatim -X PATCH \
  -F allow_merge_commit=false -F allow_rebase_merge=false -F allow_squash_merge=true \
  -f squash_merge_commit_title=PR_TITLE -f squash_merge_commit_message=BLANK \
  -F delete_branch_on_merge=true
```

With squash-only merging and linear history required, only the **PR title** (not
individual in-branch commit messages) lands on `main`. Commitizen's `release.yml`
parses `main`'s history to decide the version bump — so PR titles, not just commits,
must be Conventional-Commits-formatted. That's why `pr-title-lint` is a required check
in `ci.yml`, not an afterthought.

**The `RELEASE_PAT` secret is a hard prerequisite for "fully automatic" releases.**
With `required_pull_request_reviews` active, direct pushes to `main` are blocked for
everyone except an admin *identity*. The default `GITHUB_TOKEN` acts as the
`github-actions[bot]` identity, which is not a repo admin, so its push (the
version-bump commit + tag) would be rejected even with `enforce_admins: false`.

Resolution: Karl creates a fine-grained PAT scoped to this repo only
(`Contents: Read and write`), stores it via `gh secret set RELEASE_PAT`, and
`release.yml` authenticates with it instead of `GITHUB_TOKEN`. Since the PAT belongs to
Karl (an admin), its pushes bypass the protection rule exactly as intended.

## Execution sequence

Order matters: the repo must exist before protection can be set, and the first commit
message matters because `0.0.0` → `0.1.0` is meant to happen automatically from it.

1. Scaffold all files listed above locally — nothing GitHub-side yet.
2. `uv add --dev ruff mypy pytest pytest-cov pre-commit commitizen` → generates `uv.lock`.
3. `uv run pytest` and `uv run pre-commit run --all-files` locally until both are green
   — a sanity gate before anything touches GitHub.
4. `uv run pre-commit install` (installs both hook types via `default_install_hook_types`).
5. First commit, Conventional-Commits-formatted, e.g.
   `feat: bootstrap project tooling, CI/CD, and governance scaffolding`
   (validated by the commit-msg hook just installed). Starting `pyproject.toml` version
   is deliberately `0.0.0` so the very first automatic release run bumps it to `0.1.0`
   from this `feat:` commit — a clean first release with zero manual `cz bump` step.
6. `gh repo create verbatim --public --source=. --remote=origin --push` — the repo
   doesn't exist before this, so branch protection cannot be configured earlier. This
   first push is necessarily unprotected.
7. `gh secret set RELEASE_PAT` immediately — must exist before `release.yml`'s first
   `workflow_run` trigger fires.
8. Wait for `ci.yml` to complete on `main`; capture the exact check-run names.
9. Wait for `release.yml` to complete (triggered by CI success) — confirms the
   PAT/bypass setup actually works, producing the first `v0.1.0` tag, `CHANGELOG.md`,
   and GitHub Release, all pushed back to `main` automatically.
10. Only now apply branch protection (command above) — CODEOWNERS must already be
    present on `main`, and CI context names must already be known.
11. Configure squash-only merge settings on the repo (command above).
12. Run the verification checklist below.

## Additional recommendations

- **EditorConfig, CODEOWNERS, PR/issue templates, SECURITY.md, CODE_OF_CONDUCT.md** —
  include all (specified above).
- **CHANGELOG.md** — do not hand-seed; let the first `cz bump` in CI generate it,
  avoiding format conflicts between a hand-written stub and the tool's own output.
- **gitleaks over detect-secrets** — no baseline file for Christina to maintain.
- **Commit signing (Verified commits)** — explicitly **not** enforced in this
  bootstrap; would add setup friction disproportionate to a 2-person early-stage repo.
  Note as a future hardening option only.
- **Codecov/coverage badge** — explicitly **skip**; the `--cov-fail-under=90` gate
  inside `ci.yml` is sufficient and avoids requiring a third-party account sign-up.
- **`.github/FUNDING.yml`** — skip, not relevant.
- **CI caching** — `astral-sh/setup-uv@v5` with `enable-cache: true` for faster CI, no
  tradeoff.
- **PR-title-lint action** (`amannn/action-semantic-pull-request`) — include; the one
  addition beyond the explicit requirements list, but necessary for the squash-merge +
  automatic-versioning design to actually work.

## Verification / acceptance checklist

- [ ] `uv sync` completes cleanly; `uv run pytest` passes with coverage ≥90%.
- [ ] `uv run ruff check .` and `uv run ruff format --check .` report no issues.
- [ ] `uv run mypy src` reports zero errors.
- [ ] `uv run pre-commit run --all-files` passes every hook, including `gitleaks` and
      the raised `check-added-large-files` threshold against the existing 886 KB PRD docx.
- [ ] A deliberately malformed commit message (e.g. `git commit -m "asdf"`) is rejected
      locally by the commitizen commit-msg hook; a `feat: ...`-formatted message is accepted.
- [ ] `gh api repos/<owner>/verbatim/branches/main/protection` returns
      `enforce_admins.enabled == false`, `required_pull_request_reviews.required_approving_review_count == 1`,
      and `required_status_checks.contexts` containing the real CI/PR-title-lint check names.
- [ ] A test PR from a non-admin-simulated branch with a trivial change:
      `gh pr view --json mergeStateStatus` shows merging blocked pending review/CI.
- [ ] After one non-author approval and green CI, the PR becomes mergeable (squash
      merge, PR title enforced as commit message).
- [ ] On a second test PR, as Karl (admin), `gh pr merge --admin --squash` succeeds even
      with review/CI still pending, confirming the bypass path works.
- [ ] After any merge to `main`, `release.yml` runs automatically: `pyproject.toml`
      version increments, `CHANGELOG.md` updates, a new `vX.Y.Z` tag appears
      (`git tag --list`), and `gh release list` shows the new GitHub Release — with
      zero manual intervention.
- [ ] `git check-attr eol -- brand_guidelines.py` reports `lf`;
      `git check-attr binary -- "docs/Verbatim PRD.docx"` reports the binary attribute is
      set.
- [ ] A fresh clone by Christina, following only `README.md` (macOS or Windows),
      successfully reaches a working `uv run pytest` and active pre-commit hooks with
      no Docker/devcontainer involved.
